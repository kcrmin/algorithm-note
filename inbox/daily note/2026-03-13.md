```java
class Soldier {
	int id;
	int score;
	int ver; // 2. heap 원소에도 version 저장
}


int[] version = new int[MAX]; // 1. 현재 상태 배열 준비


// 3. 최초 등록 시 version 올리고 넣기 (init)
version[id]++;
pq.add(new Soldier(id, score, version[id]));

// 4. Update도 삭제 말고 새로 넣기
version[id]++;
pq.add(new Soldier(id, newScore, version[id]++));

// 5. 조회할 때 TOP 검사
while (!pq.isEmpty()) {
	Soldier top = pq.peek();
	if (version[top.id] != top.ver) {
		pq.poll();
		continue;
	}
	return top.id;
}

// 6. Delete는 버전 갱신하고 아무것도 안넣기
alive[id] = false; // 방법 1
version[id]++; // 방법 2: 버전 올리고 새로 안넣기
```

## **7-1. HashMap + LinkedList**
### **언제?**
- 객체를 id로 바로 찾아야 함
- 삭제를 빨리 해야 함
- 순서/그룹별로 묶어서 관리해야 함
### **장점**
- 조회 빠름
- 삭제 O(1)
- 삽입 O(1)
### **대표 문제 냄새**
- 회원 관리
- 채팅방 유저 관리
- 그룹별 목록
- 조직 관리
- 병사 목록 관리

`id로 찾고, 중간 삭제도 많다

## **7-2. PriorityQueue + Lazy deletion**

###  **언제?**
- 최대/최소를 자주 물음
- 업데이트/삭제가 많음
- heap 내부 임의 삭제가 어려움
## **장점**
- 구현 쉬움
- 삭제 안 해도 됨
## **단점**
- 쓰레기 데이터가 heap에 쌓임
- query 때 정리해야 함
## **문제 냄새**
- 최고 점수
- 가장 최근
- 우선순위 가장 높은 것
- 갱신 많음

`top만 잘 찾으면 되고, 중간 삭제는 귀찮으면`

## **7-3. Bucket**

### **언제?**
- 값 범위가 매우 작다
- 점수/등급/팀 번호 같은 범주형 값
- 15, 1100 정도
### **장점**
- 엄청 빠름
- 구현 단순
- 세그트리/PQ보다 쉬울 때 많음
## **문제 냄새**
- 점수 1~5
- 등급 A~E
- 팀 1~5
- 상태 종류가 몇 개 안 됨

`값 종류 개수가 적다`
